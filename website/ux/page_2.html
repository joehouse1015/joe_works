USABILITY TEST
介紹
Introduce
是一種用於評估產品或服務易用性的測試方法。通過將用戶置於真實或類似真實的使用場景中，測試其對產品或服務的使用體驗，以獲取有關產品或服務如何滿足用戶需求和期望的寶貴訊息。

USABILITY TEST
內部測試-介面設計
UI test
十大可用性原則是尼克森博士分析了兩百多個可用性問題而提煉出的十項通用型原則。它是產品設計與用戶體驗設計的重要參考標準，可以用來檢視產品有無符合易用性，也能夠去留意到一般使用者不會觀察到的細節。
1.狀態可見原則、2.環境貼切原則、3.使用者可控原則、4.一致性原則、5.防錯原則、6.易取原則、7.靈活高效原則、8.優美簡約原則、9.容錯原則、10.人性化幫助原則。

狀態可見原則
Visibility of system status
使用者是否知道在介面中當前操作狀態，如：所處頁面、正在處理的動作...等。

USABILITY TEST
環境貼切原則
Match between system and the real world
介面是否貼切模擬現實生活的狀況，如：介面中流程步驟以生活中實際完成步驟為準、描述的相關用詞以人們生活用語和對應角色呈現為佳，藉由模擬使用者熟悉的生活模式的介面流程對操作體驗是正面的。

USABILITY TEST
視窗
1.視窗能否基於相關的輸入或菜單命令適當的打開？
2.視窗能否改變大小、移動和滾動？
3.視窗中的數據能否用鼠標、功能鍵、方向箭頭和鍵盤操作？
4.當被覆蓋的視窗重新調用後，所有相關功能是否可操作？
5.能否使用所有視窗的相關功能，所有相關功能是否可操作？
6.相關的下拉式菜單，工具條，滾動條，對話框，按鈕，圖標和其它控制有否？能否正常顯示？完全可用？
7.顯示多視窗時，視窗名能否正確顯示，活動視窗是否加亮？
8.使用多用戶時，所有視窗是否能實時更新？
9.多次或不正確按鼠標是否會產生無法預測的結果？
10.視窗的聲音、顏色提示和視窗的操作順序是否符合需求？
11.視窗能否正確關閉？

資料
1.字母、數據能否正確顯示且輸入系統？
2.圖像方式數據(如滾動條)是否正常工作？

USABILITY TEST
下列式選單和鼠標
1.選單是否顯示在合適語言環境中？
2.應用程序的選單是否顯示系統相關特性？
3.下拉式操作是否正確，功能是否正確？
4.選單功能條是否能正常的工作？
5.能否列出所有選單功能和下拉式功能？
6.能否通過鼠標操作所有選單的功能，通過流程操作開啟選單功能？
7.選單功能隨當前狀態加亮或變灰？
8.如果要求多次點擊鼠標系統能否正確識別？
9.鼠標能否隨操作而適當改變？

USABILITY TEST
輸入介面
1. 輸入資料要完整，且要與列表資料相符合(參照數據庫進行檢查)。
2. 必填項一律在後面用*表示(必填項為空值在處理之前要有相關的提示訊息)。
3. 資料需要做檢驗，如果校驗不對需要在處理之前要有相關的提示訊息。長度、數字、字母、日期、範圍的檢驗。
4. 輸入資料的排序按照流程或使用習慣，資料特別多的時候需要進行分組顯示。
5. 下拉框不選值的時候應該提供默認值。
6. 相同資料的輸入方式應該統一(手動輸入、點選、下拉選擇、參照)。
7. 輸入後自動計算的資料要隨著別的資料修改更新(如單價變後，金額也變)。
8. 日期參照應該既能輸入，又能從輸入框選擇。
17page
USABILITY TEST
格式
1. 字體顏色、大小、對齊方式(根據資料的性質確定)、加粗的一致性。
2. 輸入框、按鈕、滾動條、列表等控件的大小、對齊、位置的一致性。
3. 所有新增、修改、查看頁面加上頁面說明(如：XXX新增、XXX編輯、XXX查看等說明字樣)，(彈出的)介面要有標題，標題與內容要一致。
4. 不同介面顯示相同資料的一致性(如列表介面和編輯介面)。
5. 介面按鈕顯示要求(搜尋、新增、刪除順序)。
6. 列表的順序排列應該統一(按照某些特定條件排序)。
7. 下拉框中的排列順序需要符合使用習慣或者是按照特定的規則排定。
8. 所有彈出視窗居中顯示或者最大化顯示。
9. 訊息列表中如果某個資料顯示過長用“…”或者分行顯示。
10. 對於帶有單位的資料，需要資料的標籤後面添加(單位)。
18page
USABILITY TEST
功能
1. 按鈕功能的實現(如返回按鈕能否返回)。
2. 訊息儲存送出後系統給出“儲存/送出成功”提示訊息，並自動更新顯示。
3. 所有有送出按鈕的頁面都要有儲存按鈕(每個介面風格一致)。
4. 凡是點選或者下拉選擇的介面，如果一旦選擇完了無法回到不選擇的情況，需要加上“清除選擇”功能按鈕。
5. 沒有選擇記錄點擊刪除/修改按鈕要提示“請先選擇記錄”。
6. 選擇記錄後點擊刪除按鈕要提示“確實要刪除嗎？”。
7. 需要考慮刪除的關聯性，即刪除某一個內容需要同時刪除其關聯的某些內容。
8. 介面惟讀的時候(搜尋、統計、導入)等，應該不能編輯。
19page
USABILITY TEST
搜尋
1. 搜尋條件是否缺少可以搜尋的字段。
2. 有些搜尋條件需要支持模糊搜尋。
3. 需要考慮有些搜尋條件本身的關聯性(即某個搜尋條件的取值範圍是依賴於其它搜尋條件的取值)。
4. 搜尋條件名稱與訊息列表及訊息編輯頁面相應的字段名稱完全統一。
5. 不同模塊相同字段的搜尋方式應該統一(手動輸入 、點選 、下拉選擇)。
6. 出報表的時候，搜尋條件需要顯示在報表標題的下面，這樣看報表的時候知道數據的依據是什麼。
7. 對於範圍的搜尋採用全閉的形式。
