介紹
Introduce
使用者訪談是使用者研究中常用的一種方法，是運用有目的、有計劃、有方法的口頭交談方式，是一種研究型的交談，是通過談話雙方的口頭交流，對有意識要獲得的資料的收集和梳理。
過程中必須力求真實，不能隨便對使用者所說的話表示贊同或者評價，並且過程中需伴隨著記錄，訪談後還要進行事後的整理和總結。

使用場景
Usage Scenarios
使用者訪談，通常適用於比較複雜的話題性場景中，例如：對經歷和過程仔細的研究，需要採集使用者的觀點以及情感等；對於復雜行為的剖析，例如：使用者是如何理解APP詳情頁各個模塊之間的關係的。
對於敏感或者私密性的話題，使用一對一的訪談法可以鼓勵被訪者願意討論；常見的定量研究，例如問卷調研和數據分析可以獲得大量的使用者行為數據，但卻無法深入了解使用者做出某種行為的具體原因和場景。
我們可以通過數據分析得知某個頁面的跳出率增加，卻無法了解到使用者這樣做的實際原因和場景。而這個時候，使用者訪談這樣的定性研究，就可以起到一定的補充作用。

Usage Scenarios

訪談形式
半結構式訪談
Semi-structured Interviews
是最常運用的一種訪談形式，雖然有事先確定的目的，但是訪談中，訪談人員可以根據訪談的進展隨時的調整問題的順序，或者新增訪談的問題等，是一種相對比較靈活的訪談形式。

訪談方法
情景訪談法
Contextual Inquiry
由Karen Holtzblatt所開發出來的訪談法，最大的特點就是，使用者把自己的體驗教給訪談人員。
一旦使用者轉化成為了教授者，那麼他就不會只講結論，而是會嘗試著講清楚前因後果。此時如果訪談人員再稍加把控，就很容易得知深層次的問題。
而我們為了讓使用者能夠不知不覺地進入教授者的角色，訪談人員必須掌握一定的技巧，這樣才能獲取有效的信息，而此時用到的最基本的技能就是引入、追問到底、重覆敘述。

USER INTERVIEW
引入
在訪談剛剛開始的時候，我們會先進行暖場提問，問幾個簡單的基礎性問題來幫助使用者快速的消除緊張感。
在這之後就要把話題慢慢的引入我們想要了解的特定主題上，進行深入的挖掘。這個過程中，需要明確的告訴使用者“我想知道的是什麼”，後續的訪談內容需要越來越聚焦到這個問題上來。

USER INTERVIEW
追問到底
情景式訪談並不是聽取和記錄使用者所說的內容就行了，只有當我們完全了解了使用者所說的話，才能算是真正的情景式訪談。
而為了做到這一點，使用者的行為和說明哪怕哪裡出現了任何不能理解的內容，訪談人員也必須刨根問底。如果內容含糊不清，就會出現訪談人員在訪談結束後根據自己的臆想詮釋使用者行為的情況。

USER INTERVIEW
追問到底-情境訪談
訪談人員：您平常什麼時候會使用銀行APP？
使用者：看薪資入帳或要轉帳繳費的時候。
訪談人員：多久會使用一次？
使用者：嗯...不太常，有需要才會用到。
訪談人員：是一個月幾次還是一個禮拜幾次呢？
使用者：大概一個月4~5次吧。
訪談人員：那可以說平均一個月5次對吧？
使用者：嗯，對的。
在訪談的過程中，使用者常常會回答像“總是”、“經常”、“不是特別高”這樣的概括與抽象的名詞，這些詞語一百個人會有一百個理解。
遇到這些回答時，訪談人員一定要盡量使用多追問，多重複的方式，盡量量化用戶回答。當然，使用者不可能向專業統計人員一樣向你提供統計信息。但是至少你需要了解使用者的“非常頻繁”是“一半以上”還是“接近90％”。

USER INTERVIEW
追問到底-常見錯誤
追問到底容易犯的一個問題就是過多的詢問使用者為什麼。並不是問為什麼不好，只是頻繁的問“為什麼”，使用者容易產生“哪來這麼多為什麼”的厭煩心理，對我們的訪談產生這樣不耐煩的情緒，我們就很難收集到真正需要的內容了。
所以可以避免的方法是，首先我們需要引導使用者對他當時的使用場景進行還原，然後了解使用者所產生的行為，在針對場景和行為去挖掘使用者產生這種體驗感受的動機，我們就能回答“為什麼”這個問題了。

USER INTERVIEW
重覆敘述
如果訪談人員已經確定了解使用者的真正意思，可以把理解的內容用自己的話敘述給使用者聽，請使用者確認是否符合他的意思。如果有理解錯誤的地方，讓使用者幫你糾正出來。
有時使用者還會根據你的敘述追加一些他當時沒有想到的關聯訊息，這樣又可以得到更多使用體驗的內容。
